# Measure API Baseline

This directory contains the implementation of the repository-owned API baseline generator.

## Layout

- `index.ts`: public facade and CLI entrypoint
- `types.ts`: shared types and the supported scenario identifier
- `config.ts`: config parsing and required-scenario validation
- `summary.ts`: median and percentile aggregation plus snapshot shaping
- `runtime.ts`: URL preflight, sequential request execution, and file writes

## What It Does

The generator does five things in order:

1. Reads and validates `apps/api/performance/baseline.config.json`
2. Fails fast if the target `/health` route is not reachable before measurement starts
3. Sends the configured number of sequential `GET /health` requests for each requested rerun
4. Reduces the collected samples into median and p95 summaries aligned with schema version 1
5. Writes the accepted snapshot JSON file back into `apps/api/performance/baselines/`

The baseline JSON file is generated output. It is committed intentionally so the repository has a stable, reviewable reference point for later comparisons. It should normally be regenerated by the script instead of hand-edited.

## What Counts As Correct

The script treats a run as correct only when these invariants hold:

1. The config parses successfully and still contains the required `api.health.get` scenario
2. The target `/health` URL responds with the expected status before measurement starts
3. Every recorded request in the accepted sample set returns the expected status code
4. Latency summary values are based on the collected request timings, not a single request
5. The saved snapshot keeps `sampleSize` as rerun count while storing internal request counts under `measurements`

Those rules are enforced partly at runtime and partly through unit tests in `apps/api/test/measure-api-baseline.spec.ts`.

## How To Extend It

If you add another API baseline lane, keep the change explicit:

1. Add or update the scenario definition in `apps/api/performance/baseline.config.json`
2. Extend `config.ts` so unsupported scenario IDs still fail fast
3. Add the scenario-specific execution logic in `runtime.ts` and any new aggregation in `summary.ts`
4. Add unit tests covering the new scenario rules or summary behavior
5. Regenerate the committed baseline JSON file and review the diff as generated output

`index.ts` should stay small. Add new behavior to the responsibility-specific files instead of growing the facade again.
