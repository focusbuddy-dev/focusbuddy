# Measure Web Baseline

This directory contains the implementation of the repository-owned web baseline generator.

## Layout

- `index.ts`: public facade and CLI entrypoint
- `types.ts`: shared types and scenario identifiers
- `config.ts`: config parsing and required-scenario validation
- `summary.ts`: median aggregation, rating rules, and snapshot shaping
- `runtime.ts`: URL preflight, Playwright/Lighthouse execution, and file writes

## What It Does

The generator does five things in order:

1. Reads and validates `apps/web/performance/baseline.config.json`
2. Fails fast if the target base URL is not reachable before any browser work starts
3. Runs the required scenarios multiple times in Playwright and Lighthouse
4. Reduces the collected samples into median-based summaries
5. Writes the accepted snapshot JSON files back into `apps/web/performance/baselines/`

The baseline JSON files are generated artifacts. They are committed intentionally so the repository has a stable, reviewable reference point for later comparisons. They should normally be regenerated by the script instead of hand-edited.

## What Counts As Correct

The script treats a run as correct only when these invariants hold:

1. The config parses successfully and still contains the required scenarios `web.home.initial-load` and `web.home.details-navigation`
2. The target URL responds successfully before measurement starts
3. The initial-load lane records `TTFB`, `FCP`, and `CLS` for every saved sample set
4. The navigation lane prefers `INP`, falls back to `FID`, and only uses `ROUTE_CHANGE_DURATION` when the browser emitted neither interaction metric
5. Snapshot summaries are based on medians across the configured run count rather than a single noisy sample

Those rules are enforced partly at runtime and partly through unit tests in `apps/web/test/measure-web-baseline.test.ts`.

## How To Extend It

If you add another baseline lane, keep the change explicit:

1. Add or update the scenario definition in `apps/web/performance/baseline.config.json`
2. Extend `config.ts` so unsupported scenario IDs still fail fast
3. Add the scenario-specific browser execution in `runtime.ts` and any new aggregation in `summary.ts`
4. Add unit tests covering the new selection rules or summary behavior
5. Regenerate the committed baseline JSON files and review the diff as generated output

`index.ts` should stay small. Add new behavior to the responsibility-specific files instead of growing the facade again.
