# import パスと関数スタイルのポリシー

本ドキュメントは Issue #96 の成果物および Issue #152 によるフォローアップ強制レビューの結果である。

目的は、ビルド / ランタイム結線が完全に固まる前に時期尚早なスタイル変更を強いることなく、アプリローカルの import パスと関数宣言スタイルに関するリポジトリの現行ポリシーを定めることである。

## スコープ

本ドキュメントが定めるもの:

- `apps/*` 内のアプリローカル import の現行既定
- 手書きコードにおける関数宣言スタイルの現行既定
- ワークスペース境界とワークスペース内コードでポリシーが異なる箇所
- これらの選好に lint 強制を導入すべきかの現行判断
- 将来エイリアス導入や強化を再検討するタイミング

本ドキュメントが定めないもの: リポジトリ全体のエイリアス移行、スタイル目的のみの広範リファクタ、スタイル選好に対する新規 lint 強制。責務指向のファイル分割やフィーチャモジュールのファイルサイズガイダンスも定めない。

## 決定サマリ

- ワークスペース間の import は、相対パスではなく、引き続きパッケージ名と公開サブパスを使う
- ワークスペース内のアプリローカル import は、現状既定で相対パスを使う
- リポジトリ全体のアプリローカルパスエイリアス標準化は、ビルド時解決・ランタイム実行・Jest 解決・ローカルコマンド・CI 強制が、ツールチェインのドリフトを起こさないほど揃うまで延期する
- 後にアプリローカルエイリアスを再検討するなら、リポジトリ全体ではなくアプリ単位で判断する
- 同一フォルダ・近傍の短い相対 import は依然として許容され、深いパスにエイリアスを採用したアプリでも既定として残る
- 手書きの export コードは、React コンポーネント・ルートハンドラ・コントローラ・マッパ・共有ユーティリティを含め、既定で「関数宣言先行」とする
- アロー関数は、ファイルローカルなコールバック・イベントハンドラ・hook コールバック・Promise チェーン・その他クロージャ志向のローカルロジックでは依然として適切である
- 実用的なメンテ価値が変更コストを上回ると明確になるまで、スタイル目的のみの移行や lint 強制は導入しない

## import パスポリシー

### ワークスペース間 import

リポジトリは既に強い境界ルールを持つ:

- アプリは共有パッケージをパッケージ名と公開サブパス経由で利用する
- アプリは他のアプリを直接 import しない
- パッケージはアプリを直接 import しない
- いずれのワークスペースも相対 import で他ワークスペースに踏み込まない

このルールは、実アーキテクチャ境界を守るため、引き続き主たる import パスポリシーである。

### ワークスペース内 import

単一アプリ / パッケージ内では、相対 import を現行のリポジトリ既定とする。

理由:

- 共有 TypeScript ベースラインは現状リポジトリ所有の `baseUrl` / `paths` マッピングを定義していない
- `apps/web` はバンドラ指向のモジュール解決で、他より早くアプリローカルエイリアスを採用しうるが、それはワークスペース固有判断であってリポジトリ標準ではない
- `apps/api` は NodeNext モジュール解決を使うため、エイリアス採用はランタイム実行・emit 出力期待・Jest 挙動・ローカルコマンド結線と併せて評価する
- リポジトリは依然として、スタイルレベルのソースパス短縮よりビルド / ランタイム結線を重視する

実務 import ルール:

- 同一ワークスペース内の手書きコードでは相対 import を使う
- 明確で安定する最短の相対パスを優先する
- パッケージ境界の import は、パッケージ名と公開サブパスのみ

後にアプリがエイリアス（例: `@/`）を採用する場合、その導入は次の条件で:

1. ツールチェイン契約を完全に検証できるアプリに限定する
2. エイリアスより明確に短い同一フォルダ相対 import を残す
3. compile / test / runtime / local-dev / CI の解決経路の更新を明示的に同梱する
4. 関連の無いパッケージや他アプリに同じスタイル判断を時期尚早に強制しない

パッケージは既定で内部ソースエイリアスを導入しない。公開エントリポイントと公開サブパスがパッケージ契約として最も重要である。

現行の検証済みアプリレベル例外:

- `apps/web` は `apps/web/src/*` を起点とする `@/` を、アプリローカルディレクトリ境界を跨ぐ際に使ってよい
- `apps/web` でも、CSS モジュールや密にローカルなヘルパなど、エイリアスより短く明確な同一フォルダ・近傍相対 import は維持する
- これは Web ローカルの許容であり、全ワークスペース既定ではない

現行の検証済みアプリレベル契約:

- `apps/api` は `apps/api` を起点とする `#api/*` をアプリローカル import に使ってよい
- 現行 API ツールチェインは、TypeScript の compile-time 解決・ソース起動・`dist` からの emit ランタイム実行・Jest 解決・Prisma コマンド実行を跨ぐ単一のリポジトリ所有契約を検証済み
- 現状、API ワークスペースで対応する唯一のアプリローカルエイリアス系統は `#api/*` である
- `@/` / `~/` / 裸の `src/*` は `apps/api` で未対応
- `#api/*` より短く明確な同一フォルダ・近傍相対 import は依然として許容する

現行の API 側モジュール解決契約は `docs/platform/api-module-resolution-contract.md` に文書化されている。

## 関数宣言スタイルポリシー

リポジトリの現行既定は「関数宣言先行」である。

対象:

- export される React コンポーネント
- Next.js のルートハンドラやその他フレームワークのエントリ関数
- NestJS の bootstrap およびアプリ向けの export 関数
- 共有マッパ・ヘルパ・ユーティリティ関数
- 手書きコード面の logger / contract / アダプタファクトリ関数

理由:

- 既存の手書きコードベースが既に名前付き関数宣言に強く寄っている
- 既定を維持することで、アーキテクチャ的利益の少ない広範な変更を避ける
- 境界の明確さ・ランタイム安全・ツールチェイン整合の方が、表現スタイルの統一より価値が高い
- 名前付き関数はスタックトレース・export・ファイル走査で識別しやすく、新たなスタイル分岐を増やさない

アロー関数は、自然に式志向となるローカル挙動で通常の選択肢である:

- イベントハンドラ
- `useEffect` などの hook コールバック / トランジション
- インライン mapping / filtering コールバック
- ファイルの主公開面に含めない、クロージャベースのヘルパ関数
- 自然に値型となるアダプタメソッドスロットやオブジェクトリテラルのコールバック実装

ローカルな式形態がより適合する既存のアロー関数を、わざわざ宣言形式へ戻す必要はない。本ルールは既定であって、スタイル純粋主義の演習ではない。

## 強制スタンス

Issue #152 は、現行ワークスペースに安定したアプリレベル import 契約が整った後にこのポリシーを再評価した。

そのレビューにより、import パスと関数スタイルの選好はドキュメント先行のスタンスを維持する。

リポジトリやワークスペースのスコープに、アプリローカルエイリアスや関数宣言形を強制する新規 lint ルールを追加しない。

本ポリシーは即時のソース移行を要求しない。

現行の強制対象は:

- ワークスペース境界の保護
- 公開サブパス規律
- ランタイム安全 / null 取り扱いルール
- 生成 vs 手書きコードの境界

レビュー後の理由:

- ワークスペース境界ルールが、アーキテクチャ的リスクのある import ミスを既に強制している
- `apps/web` / `apps/api` は検証済みのアプリローカルエイリアス契約を持つが、両者ともエイリアスより明確に短い近傍相対 import を意図的に許容している
- エイリアス使用を強制する厳格な lint は、意図的に許容している相対 import を禁じるか、メンテコストの増す例外設定を要する
- 手書き export コードは既に名前付き関数宣言に強く寄っており、新しい関数スタイル lint は既存慣習をなぞるだけで、式型コードに対するノイジーなエッジケースを生む
- ドキュメント先行のガイダンスが破綻していると示すバグクラスやレビュー差し戻しの再発も観測されていない

将来の実装作業で、特定ワークスペースに「狭く例外の少ない lint で解ける具体的なメンテ問題」が示せたなら、そのワークスペース限定の的を絞ったフォローアップとして lint を導入する。

## フォローアップ Issue の分割

本ドキュメントの判断は、リポジトリ全体の一斉清掃ではなく、焦点を絞ったフォローアップに分ける:

- #153: `apps/web` がリポジトリ既定を変えずアプリローカルエイリアスを採用すべきか検証
- #154: `apps/api` が相対 import を維持すべきか、NodeNext 契約下で安全にエイリアスを支援できるか検証
- #152: 安定したアプリレベル契約のレビューを行い、import パス / 関数スタイル選好をドキュメント先行で維持

つまり Issue #96 は判断点であって、広範な移行の実装媒体ではない。

## 本ポリシーの背後にある現状観察

- 現状、アプリ / パッケージコードの多くの手書きファイルが相対 import を使っている
- Web ワークスペースは深いアプリローカル import 向けに `@/` 許容を持ちつつ、近傍相対 import も許容している
- 現状の手書き export コードはアロー関数より関数宣言で書かれていることが多い
- API ワークスペースは compile / ソース起動 / `dist` ランタイム / Jest / Prisma の各経路で `#api/*` 契約を検証済み
- import スタイル強制は、後の Issue が具体的なメンテ価値を示さない限り、既存の境界指向ルールより弱い
- 関数スタイル強制も同様に、後の Issue が具体的なメンテ価値を示さない限り、既存の境界指向ルールより弱い

## 見直しトリガ

次のいずれかが真になったら見直す:

- 特定ワークスペースで、意図的に許容している近傍相対 import を禁じない狭い import スタイル lint で防げる繰り返しレビュー差し戻しやソースドリフトが観測される
- 関数宣言の選好を、ローカルなコールバック型コードに抑制コメントや書き換えを強いずに表現できる将来の lint
- 単なるスタイル選好を超える具体的メンテ価値を示せる将来の lint 提案

それまでは現行の低変更ルールを保つ: ワークスペース境界はパッケージ名 import、ワークスペース内 import はアプリ契約ごとに文書化、手書き export コードの既定は関数宣言、スタイル目的のみの新規 lint 強制はしない。

責務指向のファイル分割とソフトなファイルサイズガイダンスは `docs/platform/module-structure-and-file-size-policy.md` を参照する。
