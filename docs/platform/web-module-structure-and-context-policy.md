# Web モジュール構造とコンテキストポリシー

本ドキュメントは Issue #173 の成果物である。

目的は、`apps/web` の手書きコードに対するレビュー可能な実装ベースラインを定義することである。責務指向のモジュール構造、薄い公開エントリポイント、テスト配置、型安全な env アクセス、MSW サポート境界、export 意図を保つコメントに焦点を置く。

## スコープ

本ドキュメントが定めるもの:

- `apps/web` の手書きモジュールをディレクトリ / ファイルに分割する方針
- `index.ts` / `index.tsx` をいつ作り、どの程度薄く保つか
- テストの既定配置、および `apps/web/test` に残す対象
- ファイル単位の責務限界の判断基準
- Next.js のサーバ / ブラウザ / 共有 / テストコードに渡る import 境界
- 定数と環境変数の分離方針
- 副作用 / ロギング / エラーハンドリングの配置
- export コードの命名語彙
- MSW サポートコードの配置
- export された手書きコードに付けるコメント
- レビュー任せにせず自動化で強制すべきルール

本ドキュメントが定めないもの: ワークスペース横断の全体ルール、既存全 Web ファイルの一斉移行、スタイル目的のみの新規 lint。

## 決定サマリ

- 1 モジュールが意味ある関心を 2 つ以上持つようになったら、フラットな多目的ファイルではなく責務指向の feature ディレクトリを優先する
- 1 ファイルは通常 1 つの主要責務を持つ。分割はファイル長単独ではなく、ランタイム境界の混在 / 変更理由の混在 / 副作用ファミリの混在で発火する
- 公開モジュールエントリは通常 `index.ts` / `index.tsx` とし、薄く保つ
- テストは既定でソース隣接配置とし、実装と同じディレクトリに置く
- フィーチャ内の統合テストは隣接で良いが、フィーチャ横断 / フレームワークエントリの統合テストは `apps/web/test/integration` に置く
- `apps/web/test` は、ワークスペースレベルの setup、横断的な統合テスト、共有テストユーティリティのみを残す
- `constants.ts` は決定的なコード値のみ。環境変数は専用 env モジュールから読む
- 手書きコードは指定 env モジュール以外で `process.env` を直接読まない
- env アクセスは `client` / `server` / `test` のレイヤに分け、ランタイム境界に応じた呼び出し制限をかける
- Next.js のサーバ / ブラウザ境界は import の偶発詳細ではなく一級の構造判断として扱う
- MSW サポートコードは `apps/web/test/msw` ではなく `apps/web/src/testing/msw` に置く（`testing/` セグメントが非本番境界を明示する）
- テスト用フィクスチャと MSW ハンドラは、形が偶然一致しても本番ランタイム経路と共有しない
- export には短い意図コメントを付け、後の編集者がロール / 文脈をすぐ復元できるようにする
- 副作用は明示的なアクション / ハンドラ / hook / アダプタ境界に置き、view や純粋ヘルパに混ぜない
- ログは運用境界で出し、ユーザ向けエラー判断は型付きエラー情報を元に UI 境界近くで行う
- 命名は `is` / `has` / `build` / `create` / `resolve` / `fetch` / `to` のような小さな固定語彙で、export 意図をシンボル名から見えるようにする
- これらルールは可能な限り自動化で強制し、oxlint で表現できないものはリポジトリ所有の境界チェックで補う
- コメントから Issue / Discussion へリンクしてよいが、詳細なプロダクト推論はソースコードに含めない

## モジュール構造ルール

### 責務先行ディレクトリ

1 つの手書きモジュールが意味ある関心を 2 つ以上持つようになったら、薄い公開エントリポイントを持つディレクトリを優先する。

意味ある関心の分割例:

- 公開エントリポイント vs 内部実装詳細
- React の view 構成 vs 派生表示データ
- ランタイムロジック vs 定数
- ランタイムロジック vs テストフィクスチャ
- API 面 vs 内部マッピングヘルパ

例（`web-baseline-page` ディレクトリ）:

```text
web-baseline-page/
  index.tsx
  view.tsx
  sections.tsx
  constants.ts
  types.ts
  index.test.tsx
```

ファイル名は変えてよいが、分割は責務に従う。任意の行数を達成するために分割しない。

### 1 ファイル 1 主要責務

「1 ファイル 1 関数」ではない。読み手が「このファイルが所有する主な仕事は何か」に答えられること。

通常分割するのは:

- export メンバが異なる理由で変わる
- サーバ専用 / ブラウザ専用 / テスト専用が混在
- 純粋整形と外部副作用が同居
- `view` / `mapping` / `actions` / `fixtures` のように独立命名できるセクションが複数ある
- 1 つの変更を理解するのに無関係セクションを読み飛ばす必要がある

ヘルパが複数あるだけでは分割の理由にならない。同じ主要責務に資する小ヘルパは同居してよい。

### ソフト分割シグナル

絶対上限ではなくレビューシグナルとして扱う:

- 1 ファイル概ね 150 行
- 1 ファイル内に 2 つを超える export ランタイム面
- 副作用ロジックと presentational マークアップの同居
- 本番データとテストフィクスチャの同居

### 薄いエントリポイント

`index.ts` / `index.tsx` は公開エントリポイントであり、保管庫ではない。通常以下のみを行う:

- 主実装の export
- 小さな依存結線
- ディレクトリの最小公開面

### バレル（barrel）ルール

許容するバレル様エントリは:

- 1 つの安定した公開面を露出する feature / モジュールの `index.ts` / `index.tsx`
- ワークスペース契約の一部であるパッケージ export 境界

`common/index.ts` / `constants/index.ts` のような広い利便性バレルは避ける。

## import 境界ルール

### 4 つのランタイムクラス

`apps/web` の手書きソースは次の 4 クラスで考える:

- サーバエントリ
- ブラウザエントリ
- 共有純粋
- テスト専用

### サーバエントリ

例: `'use client'` の無い `src/app/**` のサーバコンポーネント、ルートハンドラ、リクエストデータ / サーバ専用 env を読むサーバヘルパ。

import してよい:

- 共有純粋
- サーバ専用アダプタ / env モジュール
- フレームワークがサポートする境界経由でのみブラウザコンポーネント

import してはならない:

- `apps/web/src/testing/**`
- テストフィクスチャ
- DOM グローバル前提のブラウザ専用ランタイムヘルパ

### ブラウザエントリ

例: `'use client'` を含むモジュール、ブラウザ専用イベント / エフェクト制御。

import してよい:

- 共有純粋
- ブラウザ安全なアダプタ / hook
- `client` env アクセサ（`public` のみ）

import してはならない:

- サーバ専用 env モジュール
- `next/headers` / `next/cookies` のリクエスト専用ヘルパ
- Node 組み込み / サーバ専用ランタイムアダプタ
- `apps/web/src/testing/**`

### 共有純粋

サーバ / ブラウザのいずれからも安全に使えるコード。

import してはならない:

- `next/headers` / `next/cookies` 等のリクエスト専用 API
- DOM グローバル / ブラウザ専用モジュール
- Node 組み込み
- `apps/web/src/testing/**`
- 意図的に共有された型付き値モジュール以外の env モジュール

これらが必要なヘルパは共有純粋ではない。サーバ / ブラウザ / テスト境界へ移す。

### テスト専用

`apps/web/src/testing/**` および `apps/web/test/**` 配下に置く。

検証目的で本番モジュールを import してよいが、本番ランタイムコードはテスト専用コードを逆 import しない。

### プレビューデータ vs フィクスチャ

シップされるランタイムが使うプレビューデータは、見た目が偽データでも本番所有のデータである。

テスト / MSW 用のフィクスチャは、見た目が現実的でもテスト所有のデータである。

形が一致するからといって 1 つのヘルパに統合しない。

## テスト配置ルール

### ソース隣接が既定

1 モジュールのテストは既定でソース隣接配置とする。

例: `index.test.ts` / `index.test.tsx` / `view.test.tsx` / `client.test.ts`。

レビュアが 1 つのディレクトリを開いた時点で、テストの有無が即座に分かる状態を目指す。

### 統合テストの配置

複数の内部ユニットの協調を検証する場合は `*.integration.test.ts` / `*.integration.test.tsx` を使う。

- 1 feature ディレクトリ内に収まる協調なら隣接配置のままでよい
- 次の境界を跨ぐなら `apps/web/test/integration/**` に移す:
  - 複数フィーチャを跨ぐ
  - ルートハンドラ・ミドルウェア・app-router 結線などフレームワークエントリ
  - 1 ソースモジュールより広い共有 setup / ヘルパ

### テスト命名

- ユニット / モジュールローカル: `*.test.ts` / `*.test.tsx`
- 統合: `*.integration.test.ts` / `*.integration.test.tsx`
- ランナー setup: `setup-tests.ts`

`constants.ts` / `helpers.ts` のような名前で本番に偽装したテストサポートを置かない。

### `apps/web/test` に残すもの

- Jest setup（例: `setup-tests.ts`）
- 多モジュールで意図的に共有するテストユーティリティ
- 1 ソースディレクトリより大きい横断統合テスト
- 1 ソースモジュールが自然な置き場でないフレームワーク級テスト

`apps/web/test/integration` / `apps/web/test/utils` のようなサブディレクトリを使い、フラットなトップレベルにしない。

## 定数と環境ルール

### `constants.ts` は決定的な値のみ

ラベル / メッセージテンプレ / 固定数値しきい値 / ルートセグメント定数 / フィーチャローカルな表示既定など、安定したコードレベルの事実に使う。

env アクセス・ランタイム機能検出・テストフィクスチャの混在ダンプ場にしない。

### 配置

1 フィーチャのみで意味を持つ値はフィーチャローカルな `constants.ts` に置く。

複数フィーチャで再利用する Web ローカル定数は具体的な共有パスへ:

- `src/common/constants/routing.ts`
- `src/common/constants/display.ts`
- `src/common/constants/<domain>.ts`

巨大な共有定数ファイルを作らず、トピック別に分ける。

### コピーは原則ローカル

1 コンポーネント / 1 ルートを説明するための短い文言は、当該モジュールにローカルに置く。

共有定数へ移すのは:

- 同じ文言を複数モジュールで意図的に再利用するとき
- ルートセグメント / ステータスラベルのような安定した意味契約の一部のとき
- 一元化が読みやすさより明らかにドリフト軽減に勝るとき

### フィクスチャは定数ではない

サンプルペイロードやテスト既定値は本ポリシー上の「定数」ではない。テスト / プレビュー専用モジュールに置く。

### env アクセスは型付き / 集中化

env アクセスは専用モジュールから:

- `env/client.ts`
- `env/server.ts`
- `env/test.ts`

契約:

- `process.env` の読み取りは env モジュール境界内に限る
- そこで検証 / 正規化する
- 型付き値 / 型付きゲッタを export する
- アプリコード / コンポーネント / ルートハンドラ / 共有ランタイムヘルパからは `process.env` を直接読まない

### env レイヤの呼び出し制限

- ブラウザコードは `client` のみ import 可
- サーバコードは `client` / `server` を import 可
- テストファイル / setup は `test` ヘルパを import 可
- 本番コードは `test` ヘルパを import しない

複数レイヤで同じ値が必要なら、別レイヤに踏み込ませず、適切なモジュールから意図的に export する。

## MSW 構造と境界ルール

### 既定配置

```text
apps/web/src/testing/msw/
  handlers/
  factories/
  browser.ts
  server.ts
```

`apps/web/test/msw` より優先するのは MSW がテストファイルを超えた責務を持つため。`apps/web/src/msw` より優先するのは `testing/` セグメントが非本番境界を明示するため。

### 本番境界ルール

本番ランタイムコードは `src/testing/*` から import しない。対象は `src/app` / `src/components` / `src/lib` 配下の本番ランタイムヘルパなど。

ベースラインや空状態体験のためのプレビューデータが必要なら、MSW factory やテストフィクスチャからではなく、本番所有の demo / preview モジュールから供給する。

### フィクスチャ分離ルール

テストフィクスチャとランタイムプレビューデータが似通っていても、同一プロダクト所有契約を真に表す場合を除き別モジュールに分ける。

## export コメントルール

### 既定

手書きの export 関数 / コンポーネント / クラス / 定数オブジェクト / 公開モジュール面には、ロールを説明する短い先頭コメントを付ける。

### 固定形式

推奨順:

- `Role:` この export が所有するもの
- `Boundary:` 利用可能箇所、または依存してはならないもの
- `Ref:` 履歴が重要なら任意の参照

例:

```ts
/**
 * Role: Resolves the browser-safe API base URL for web client calls.
 * Boundary: Public env only. Must not read server-only env values.
 * Ref: #173
 */
export function getPublicApiBaseUrl() {}
```

`Ref:` は判断が arbitrary に見える場合のみ。テンプレ充足のための空フィールドは入れない。

### 例外

ファイルレベルコメントが既に明示しているなら、繰り返しの per-export コメントは不要:

- 1 つの小さな types ファイル内の密接に関連する export 型
- 純粋な再 export バレル
- 生成コード
- フレームワーク要請の export（`metadata` / `config` / route メソッド等）でファイル先頭コメントが境界を説明済みのもの

### 外部参照

リポジトリ外の文脈は、コメントに長文を写さず短いリンクで参照する。

## 副作用配置ルール

### 中核の分割

- view はデータを描画する
- アクション / ハンドラが副作用をトリガする
- アダプタが外部システムに触れる
- 純粋ヘルパが値を整形する

### view

presentational コンポーネントと純粋描画ヘルパ。所有しない:

- ネットワーク呼び出し
- ルータナビゲーション
- ロガー出力
- ヘッダ / Cookie 直接アクセス
- ストレージアクセス
- 時間ベース制御 / サブスクリプション

view はコールバック・データ・準備済みステートを受け取る側に留める。

### アクション / ハンドラ / hook

ブラウザ側副作用は明示的境界に置く: イベントハンドラ、カスタム hook、コントローラ様アクションヘルパ。

これらが所有してよい:

- `router.push`
- ミューテーション送信
- ロガー出力
- ストレージ書き込み
- サブスクリプションのセットアップ / クリーンアップ

`useEffect` は禁止ではないが、純粋派生や一般ビジネスロジックを隠すためではなく、実ライフサイクル同期に使う。

### アダプタ

API クライアント / リクエストリーダ / Cookie / ヘッダリーダ / ブラウザストレージラッパ / ロギングランタイムアダプタなど。型付き値を返し、境界固有詳細を view に漏らさない。

## ロギングとエラーハンドリングのルール

### ロギングの目安

運用シームを跨ぐ、または復旧判断を下す境界でログを出す:

- ルートハンドラ
- ミドルウェア
- サーバアクション
- リモート呼び出しが失敗した API アダプタ
- ユーザに見える復旧経路を選んだブラウザ側アクションハンドラ

同じ失敗を全層で重複ロギングしない。下層が権威ある運用ログを出したなら、上層はログを重ねずユーザ向け処理に集中する。

### エラー変換

エラーを静かに飲み込まない。`catch` してよいのは次のいずれかのとき:

- 型付き結果 / 共有エラー形に変換する
- 境界文脈を付与して再 throw する
- Web エラーポリシーが明示的に許す UI 向けフォールバックを選ぶ

### ユーザ向け vs 運用者向け

- ユーザ向けコードはリダイレクト / インラインフィードバック / リトライ / Not Found / ルート致命の挙動を決める
- 運用ログは request ID / 境界文脈 / サニタイズ済み技術詳細を残す
- 復旧判断は [Web エラーハンドリングポリシー](./web-error-handling-policy.md) を参照する
- ロギング配置は [ロギングと監査の境界](./logging-and-audit-boundaries.md) と整合させる

## 命名ルール

実装形ではなく意図を表す名前を使う。

- `is` / `has` / `can` / `should`: bool 返却の判定
- `build`: 既知入力からの純粋アセンブリ
- `create`: 状態 / ランタイム設定を保持しうる依存結線 / ファクトリ
- `resolve`: 入力 / 文脈を 1 つの判断に解釈
- `get`: 同期 / 外部 I/O 無しの単純取得
- `read`: リクエスト / ストレージ / 他のスコープ付きソースからの境界読み取り
- `fetch`: ネットワーク / リモート I/O
- `load`: 1 呼び出し用に複数の read / fetch を組み合わせる制御
- `to`: 別表現への純粋変換
- `parse`: 外部入力の検証 / 正規化
- `format`: 表示 / ログ用の文字列描画
- `assert`: 失敗時に投げる不変条件チェック

`handleData` / `processThing` / `utils` のような曖昧名は、より狭い動詞が使えるなら避ける。

## 強制と自動化のルール

### 強制スタンス

ドキュメント止まりにせず、ツーリングが許す範囲で fail-fast の自動化を優先する。

### 最初の自動化対象

- `src/env/**` 以外での `process.env` 直接アクセス禁止
- `apps/web/src/testing/**` からの本番 import 禁止
- ブラウザエントリでのサーバ専用 env / リクエスト専用ヘルパ import 禁止
- `*.test.*` / `*.integration.test.*` の命名強制
- 許容された `index.ts` / `index.tsx` 境界以外での広い利便性バレル禁止

### ツーリング分割

直接表現できるなら oxlint を使う。表現できないものはリポジトリ所有の境界スクリプト / AST チェックで補う:

- 手書きランタイム面の export role コメント
- 1 ワークスペース内のパスベースなランタイム境界ルール
- ファイル名 / ディレクトリ役割に基づく狭いバレル制限

## レビューチェックリスト

- フィーチャに複数責務があるとき、薄い公開エントリポイントが露出されているか
- 各ファイルが依然として 1 つの主要責務を持つか
- モジュールローカル挙動のテストがソースディレクトリから可視か
- 横断的な統合テストは `apps/web/test/integration` 配下にあり、ユニットテスト場所に混在していないか
- import がサーバ / ブラウザ / 共有 / テストの境界を尊重しているか
- 定数が env アクセス / フィクスチャから分離されているか
- 共有定数がトピック別に分かれているか
- `process.env` が型付き env モジュール経由のみか
